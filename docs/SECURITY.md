# 安全说明

## 支持范围

只维护 `main` 和最新 GitHub Release。旧版本发现安全问题时应先升级，再确认问题是否仍然
存在。

## 报告问题

安全漏洞优先通过 GitHub 的私密 Security Advisory 报告。请提供：

- 受影响版本和 Linux 环境；
- 最小复现步骤；
- 预期与实际结果；
- 是否需要认证、读权限或写权限；
- 日志中的请求 ID。

不要提交真实密码、Authorization、Cookie、私有文件内容或主机密钥。尚未确认修复前不要
公开可直接利用的细节。

## 部署边界

Ram 只提供 HTTP/1.x，不提供 HTTP/2。域名、私有 TLS 和证书由网关或反向代理负责。Ram 不信任
`X-Forwarded-For` 等来源头，来源限流只使用直接 TCP 对端。

最低部署要求：

- 使用独立、低权限的 Linux 服务账号；
- 只把监听端口暴露给受信任网关或内网；
- 启用 Basic 或 Digest 认证；Basic 优先使用 Argon2id，Digest 使用独立的强明文密码；
- 配置和认证文件位于服务目录之外，权限最小化；
- 一个可写服务目录只运行一个 Ram 实例；
- 使用本地、语义可靠的文件系统；
- 由 journald 或容器运行时保管和轮转日志；
- 定期备份并演练恢复。

## 保留的安全属性

- Linux `openat2`/dirfd 根目录隔离；
- `RESOLVE_BENEATH`、magic-link 和跨挂载防护；
- 特殊文件、FIFO、socket 和设备文件拒绝；
- 服务目录与配置、认证等敏感路径隔离；
- PUT 私有候选、原子发布、失败清理、`fsync`；
- ETag、条件请求、Range 和并发覆盖防护；
- 请求头、请求体、目录、搜索、ZIP 和响应大小限制；
- 连接、请求、来源、用户、上传和昂贵任务并发限制；
- Basic/Digest 失败退避和认证限流；
- CSP、no-store、nosniff 等安全响应头；
- 凭据和敏感查询参数日志脱敏；
- 超时和优雅关闭。

## 不提供的保证

Ram 不防御拥有 root、服务账号或服务目录直接写权限的本地攻击者；不协调多个进程共同
写入；不扫描恶意文件；不提供端到端加密；不把隐藏列表当作访问控制；也不保证网络文件
系统、FUSE 或运行中被特权进程改变的挂载拓扑。

更完整的假设见 [THREAT_MODEL.md](THREAT_MODEL.md)。
